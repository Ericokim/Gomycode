# Reflection Report

The main challenge during the schema refactor was balancing requirements that do not all need the same data model. The first design focused on product browsing and reliable order storage, so a document database worked well. Products could be searched and filtered, while orders could embed customer and item snapshots for fast reads and historical accuracy. However, the new analytics and high availability requirements made the original design incomplete because a checkout-focused schema is not always efficient for large sales reports or high-volume event storage.

The new requirements changed the design by forcing a separation between operational and analytical workloads. I added sharding so product and order data can be distributed across nodes, replication so the system can stay available during failures, and denormalized analytics collections so reports do not need to scan every order. I also introduced a wide-column event table for order events because analytical data is append-heavy and needs to handle high write throughput.

The refactor improves scalability by spreading reads and writes across shards and replicas. It improves availability because replicated data can still be served if a node fails. Query performance improves because customer-facing queries use optimized operational collections, while analytics queries use pre-aggregated summaries such as daily sales by product and category. The main trade-off is complexity. Denormalized data must be updated carefully, analytics can become eventually consistent, and the system now uses more than one NoSQL model. Overall, the refactored design is stronger because each workload has a schema that matches its access pattern.
